Layer 1

Physical underlay

  • Datacenter fabric
  • Upstream routers and BGP peers
  • Routed VLAN-backed networks
  • MTU, DNS, NTP, IPAM
  • Certificate authority and firewall policy

Layer 2

VCF platform

  • Management domain first
  • One or more workload domains
  • Consolidated or workload-hosting model
  • Tenancy and scale boundaries

Layer 3

NSX services

  • NSX Manager
  • Transport zones
  • Overlay and VLAN segments
  • Edge cluster, Tier-0, Tier-1
  • NAT, service insertion, north-south path

Layer 4

Application delivery

  • Avi Controllers
  • Service Engines
  • L4/L7 virtual services
  • Health monitors and policies
  • App-facing VIP delivery

Recommended reference pattern

  • Management domain with SDDC Manager, vCenter, NSX Managers, Edge cluster, shared services, and Avi Controllers
  • Workload domains for app clusters, optional Supervisor workloads, and tenant segments behind Tier-1
  • Dedicated or shared Service Engine Groups by environment

Core design principle

  • VCF owns domain lifecycle
  • NSX owns routing, segmentation, and stateful network services
  • Avi owns application delivery and modern load balancing
  • Shared standards own DNS, certs, logging, HA, and operations

Build sequence

  1. Confirm VCF version, scale, tenancy, app classes
  2. Design underlay routing and IP plan
  3. Lay out management and workload domains
  4. Design NSX topology
  5. Design Avi placement and Service Engine groups
  6. Define app patterns, security, HA, then HLD and LLD

Reference patterns

Single-site enterprise baseline

  • 1 management domain
  • Shared Edge cluster
  • 3 Avi Controllers in management domain
  • Separate workload domains by environment or business boundary
  • Dedicated VIP pools per environment

Shared platform services pattern

  • Central shared services in management or shared platform domain
  • Shared DNS/IPAM, certs, logging
  • Shared Service Engine Group for platform use only
  • Separate app-facing service groups for tenant traffic

Supervisor-aware pattern

  • Define ingress ownership up front
  • Separate internal and external service classes
  • Explicit namespace or tenant boundaries
  • Cluster-to-Service Engine connectivity rules

Brownfield coexistence pattern

  • Inventory legacy VIPs and monitors first
  • Keep rollback path per wave
  • Separate uplift tasks from service migration
  • Document support boundaries for old and new paths

Hybrid cloud extension pattern

  • Private cloud remains control point
  • Use cloud landing zones for extension, not sprawl
  • Map services by capability domain
  • Keep ownership explicit across clouds

Service class pattern

  • Internal application VIPs
  • External/public VIPs
  • Kubernetes/Supervisor services
  • Shared platform services

HA / DR

Platform HA

  • Redundant management-domain infrastructure
  • Multiple NSX Managers
  • Redundant Edge nodes
  • Resilient shared platform services

LB control-plane HA

  • 3-node Avi Controller cluster
  • Dedicated control-plane health monitoring
  • Admin access and break-glass model
  • Backup and restore tested

LB data-plane HA

  • Multiple Service Engines
  • Service Engine Groups aligned to failure domains
  • Frontend VIP redundancy
  • Deterministic routing and return path

HA / DR validation matrix

Scenario What to test Expected result
Controller lossController cluster resiliency, admin continuity, policy retentionManagement recovers without uncontrolled service impact
Service Engine lossData-plane failover and health-monitor responseVIP traffic fails over within documented behavior
Edge lossTier-0/Tier-1, north-south routing, NAT dependencyRouting and stateful services remain within HA design
Upstream route failureBGP resiliency and alternate path behaviorNorth-south traffic converges as designed
Certificate failureExpired or rotated certificate handlingDocumented alerting and rotation process works

Security blueprint

Security controls to define up front

  • Dedicated management networks for VCF and Avi control plane
  • Frontend VIP separation from backend app networks
  • TLS termination policy by service class
  • Certificate lifecycle ownership
  • DFW and segmentation policy for Controllers, Service Engines, apps, and management services
  • Administrative RBAC split among VCF, NSX, and LB teams

Advanced app-security options

  • WAF-required service classes
  • Internet-facing service hardening
  • Logging and analytics per app class
  • Segmentation alignment with vDefend and broader workload controls

Kubernetes / Supervisor blueprint

What to define

  • Ingress ownership
  • L4 versus L7 service boundaries
  • Cluster-to-Service Engine connectivity
  • Tenant or org separation
  • DNS zone strategy for app teams

Supervisor-aware design notes

  • Keep platform-team and app-team responsibilities explicit
  • Separate internal and external service classes
  • Define cert ownership per ingress model
  • Decide whether namespaces or orgs map to distinct service boundaries
